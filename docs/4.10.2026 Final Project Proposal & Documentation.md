## Project Proposal

### Project Abstract

The Vault Keeper is a self-hosted cloud storage platform designed to provide data independence by eliminating 3rd party data mining and providing free customization. While products like Google Drive and Dropbox already exist, users don't have full freedom to customize and use these services. With "The Vault Keeper," I propose a lightweight and open-source solution that allows users to customize and manage their own data for free. The software's main use is to provide an environment that allows users to upload, store, and manage their files on a private Linux server. The file management will be seamless and provide log file data to track changes to files. This software is intended for people who care more deeply about their digital privacy, developers looking for secure and internal file sharing, and users looking for completely free and customizable alternatives to existing products. This project demonstrates good security practices and offers users a secure environment for internal data handling.

### High Level Requirement

The Vault Keeper allows users to securely access a private storage environment to upload, manage, and download personal files through an intuitive web interface. The system ensures data privacy by isolating storage on an Ubuntu Linux server and maintains transparency by automatically logging all file interactions in an activity log.

### Conceptual Design

The Vault Keeper is built on a high-performance, client-server architecture designed for privacy and environmental isolation. The system is split into a zero-knowledge frontend and a high-concurrency backend to ensure that sensitive data is encrypted before it ever leaves the user's local environment.

The system uses a model where the storage backend resides on a private Ubuntu Linux server, isolated from the user's primary hardware to protect against local failures or malware.

- **Frontend:** All encryption happens client-side. The server acts as a vault that stores encrypted files without ever possessing the keys to read them.
- **Backend:** The backend manages the intake of these encrypted files by organizing the metadata in the database and handles the activity log auditing.

With the decision to prioritize performance and modern web standards, the following tools have been selected:

| Category | Tool | Reasoning |
|---|---|---|
| **Frontend** | React (Vite) + Tailwind | Enables fast development, easy Dark Mode implementation, and PWA support. |
| **Backend** | FastAPI (Python) | Selected over Flask for its high performance with file streams and greater asynchronous support. |
| **Database** | SQLite | A simple, portable, and file-based database that requires no extra server overhead. |
| **Storage** | Local Linux Directory | The most direct way to handle large file volumes on a self-hosted Ubuntu server. |
| **Auth/Security** | Web Crypto API | Implements client-side encryption to ensure the server remains zero-knowledge. |

### Proof of Concept

- **Repository:** https://github.com/cis3296s26/The-Vault-Keeper
- **Live Beta:** https://the-vault-keeper.vercel.app/

This proof of concept was done on a Windows environment to check framework compatibility. Currently, a live beta is deployed using Vercel to demonstrate the React/FastAPI integration. In the final project, the Ubuntu Server will serve as a separate, standalone deployment environment. With this key element, the application logic and data storage are isolated from the user's machine, providing environmental isolation. A `README.md` is included with System Specifications & Compatibility as well as instructions on how to run.

### Background

The Vault Keeper addresses concerns with cloud platforms and data ownership, with similar platforms like Google Drive lacking transparency. While there are existing alternatives, they are often too complex for a single user's needs. The Vault Keeper is a project designed to be a simpler, lightweight version of these existing services. By building this from scratch with free tools, I am proposing a Client-Server architecture where the Ubuntu Linux Server acts as a separate deployment environment from the user's local machine. Using FastAPI to manage this architecture allows for efficient data isolation, protecting files from hardware failures or malware. This provides users with a personal and transparent way to manage their data across a network rather than storing it on a local hard drive.

### Required Resources

- **Software & Frameworks:** Python 3.10+ with FastAPI for the backend and React (Vite) for the frontend.
- **Infrastructure:** Ubuntu Server 22.04 LTS hosted via VirtualBox for environmental isolation.
- **Data & Security:** SQLite for metadata/logs and Web Crypto API for client-side encryption.
- **Hardware & Tools:** GitHub for version control and a PC/Laptop with at least 8GB RAM to handle the development tools and VM simultaneously.

## Project Design

### Vision

For everyday people who care about what happens with their personal data and want to take control of how that data is managed. The 'Vault Keeper' provides a customizable application that allows users to track and manage personal files on a free and private Linux server. Unlike traditional file managers that store these files for the user, 'The Vault Keeper' is solely managed by the user. Our product uses this advantage to provide security to the user, as they know that they are in control and third parties won't have access to their data.

### Class Diagram

> *(See `Final_Project_Diagram.jpg` in the project files)*

**Explanation:**

This class diagram shows the four main parts of the app and how they connect:

1. **User** — Stores the username and password and handles logging in and out.
2. **File** — Represents anything that's uploaded and tracks the name, date, size, and handles encrypt/decrypt operations.
3. **Activity Log** — Keeps a record of changes to a file. The filled diamond connection signifies that a log entry cannot exist without its file; if the file is deleted, so is its history.
4. **Vault Manager** — The coordinator that doesn't store anything on its own but calls core commands such as `uploadFile()`, `deleteFile()`, and `fetchLogs()`.

### Sequence Diagram

> *(See `Final_Project_Diagram.jpg` in the project files)*

**Explanation:**

This sequence diagram shows the main sequence of events between the different parts of the application:

1. The User clicks upload and the Frontend receives the file.
2. The Frontend encrypts it using the Web Crypto API.
3. The encrypted file is sent to the Backend.
4. The Backend checks that the user is authenticated.
5. The file gets saved to Storage.
6. The Backend logs the action in the Database and a confirmation of success is returned to the User.
